## Why The Original 3/7 Happened

For the 2026-03-18 authoritative run, the key point is that the 4 unapplied programs were not all “0 sites”:

1. `event_exit_acct_process`
   - Live enumerate finds 6 branch-flip sites.
   - The static object scan of `corpus/build/tetragon/bpf_exit.bpf.o` reports 0 sites.
   - Result: no policy file is generated/resolved, so the program stays on stock.

2. `tg_kp_bprm_committing_creds`
   - Live enumerate finds 3 branch-flip sites.
   - The static object scan of `corpus/build/tetragon/bpf_execve_bprm_commit_creds.bpf.o` reports 0 sites.
   - Result: no policy file is generated/resolved, so the program stays on stock.

3. `execve_send`
   - Live enumerate finds 5 branch-flip + 9 cmov.
   - The old policy file was effectively empty because the static scan only surfaced cmov and the default generator skips cmov.
   - Result: policy remap selects 0 live sites, so recompile is skipped as a noop.

4. `execve_map_update`
   - Live enumerate finds 1 branch-flip.
   - The old policy file was also empty because the static scan only surfaced cmov.
   - Result: policy remap selects 0 live sites, so recompile is skipped as a noop.

So the real answer to “why only 3/7?” is:

- 2 programs were missing policy coverage because static-object scanning missed live-only branch-flip sites.
- 2 programs had policy files, but those policies were empty because they were generated from stale/static cmov-only views.

This is a policy-coverage problem first, not a “0 sites” problem.

## Bottom Line

### Root cause for the original unapplied programs

The original 4 unapplied cases were mainly policy coverage misses:

- `event_exit_acct_process`: missing policy because static scan saw 0 sites, live scan saw 6 branch-flip.
- `tg_kp_bprm_committing_creds`: missing policy because static scan saw 0 sites, live scan saw 3 branch-flip.
- `execve_send`: empty policy because static scan was cmov-only, live scan had 5 branch-flip.
- `execve_map_update`: empty policy because static scan was cmov-only, live scan had 1 branch-flip.

### How far I got

- In isolated VM probes, I proved:
  - `event_exit_acct_process` can apply.
  - `execve_send` can apply.
  - `execve_map_update` can apply in isolation.
- In a reordered live session, I proved `5` Tetragon programs can apply together on the current kernel before workload execution:
  - `event_exit_acct_process`
  - `event_wake_up_new_task`
  - `event_execve`
  - `execve_map_update`
  - `execve_send`

### What still blocks a clean E2E win

- `execve_rate` is now a current-kernel validator/kernel reject.
- `tg_kp_bprm_committing_creds` also becomes a current-kernel reject once policy coverage is added.
- After the benchmark workloads have exercised the programs, expanding the recompiled set causes a live-recompile instability before I can collect post-ReJIT metrics.

## Recommended Next Steps

1. Treat policy coverage and kernel validity as two separate issues.
   - Policy coverage is now understood.
   - The remaining blockers on the current kernel are validator/runtime issues.

2. Root-cause the current-kernel rejects first.

4. Once the kernel-side instability is fixed, rerun the same VM baseline and after-run on the 2026-03-20 kernel.
   - The no-workload combined probe already shows the coverage upside is real.
