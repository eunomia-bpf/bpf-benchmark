# v5 Design Gap Analysis

Date: 2026-03-10

This review compares the current v5 code against the v5 design, not against the earlier minimal-POC scope. The implementation is materially better than the first 4.5/10 review: overflow-safe site-range helpers are now in place, CPU legality is partly derived in-kernel, `COND_SELECT` is on the v5 userspace path, and the v5 scanner/blob builder now lives in `scanner/` rather than `micro/runner` (`vendor/linux-framework/kernel/bpf/jit_directives.c:562-580, 583-592, 1638-1657`; `scanner/src/pattern_v5.cpp:340-756`; `micro/runner/src/kernel_runner.cpp:767-816`). The core design gap, however, is still the same one called out in the plan as blocker #58: the binding-table / parameterized-emitter half of v5 is still missing (`docs/kernel-jit-optimization-plan.md:735-739`).

## 1. Feature Matrix

| # | Design requirement | Current status | Gap description |
|---|---|---|---|
| 1 | **Pattern descriptor format**: variable-length rule with pattern entries carrying `opcode`, `opcode_mask`, register/immediate fields, wildcard support (`docs/tmp/jit-pass-framework-v5-design.md:166-235`, `docs/tmp/jit-pass-framework-v5-design.md:813-839`) | **Partial** | The blob is variable-length in the implemented v2 ABI, and each rule does inline a pattern array plus constraints (`vendor/linux-framework/include/uapi/linux/bpf.h:1588-1600`; `vendor/linux-framework/kernel/bpf/jit_directives.c:1863-1962`). But the actual pattern entry is `struct bpf_jit_pattern_insn { __u8 opcode; ... }`, not `opcode_mask` + `opcode_value` (`vendor/linux-framework/include/uapi/linux/bpf.h:1565-1576`). The matcher does exact `insn->code != pattern->opcode` comparison in both kernel and scanner (`vendor/linux-framework/kernel/bpf/jit_directives.c:1512-1513`; `scanner/src/pattern_v5.cpp:220-221`). Wildcards exist only for dst/src/imm/off by leaving bindings unset or exact-check flags clear; opcode-class wildcards from the design are missing. |
| 2 | **Binding slots / variables**: pattern variables bind values, then a binding table maps them to canonical emitter parameters (`docs/tmp/jit-pass-framework-v5-design.md:589-702`) | **Partial** | Pattern-variable binding exists: the kernel matcher fills `vars[]` with typed REG/IMM/OFF values and cross-checks reuse (`vendor/linux-framework/kernel/bpf/jit_directives.c:1474-1534`); the scanner mirrors the same logic (`scanner/src/pattern_v5.cpp:122-253`). What is missing is the second half of the design: there is no `binding_count`, no `struct bpf_jit_binding`, no binding pointer in kernel-internal `struct bpf_jit_rule`, and no canonical-param array stored on a rule (`vendor/linux-framework/include/uapi/linux/bpf.h:1588-1600`; `vendor/linux-framework/include/linux/bpf_jit_directives.h:54-81`). As a result, emitters never consume bindings. |
| 3 | **Constraint system**: generic post-match constraints such as SAME/DIFF, ranges, etc. (`docs/tmp/jit-pass-framework-v5-design.md:272-343`) | **Implemented** | The implemented constraint set is generic and close to the design: `EQUAL`, `SUM_CONST`, `IMM_RANGE`, `NOT_ZERO`, `MASK_BITS`, `DIFF_CONST`, and `NOT_EQUAL` are in the UAPI and enforced in one kernel interpreter (`vendor/linux-framework/include/uapi/linux/bpf.h:1555-1563`; `vendor/linux-framework/kernel/bpf/jit_directives.c:1540-1612`). Because bindings are typed, `EQUAL` / `NOT_EQUAL` effectively cover SAME_REG / DIFF_REG style checks for registers, immediates, and offsets of the same type (`vendor/linux-framework/kernel/bpf/jit_directives.c:1474-1493`). This part of the design is substantially there. |
| 4 | **Canonical forms**: a small semantic vocabulary (`SELECT`, `WIDE_LOAD`, `ROTATE`, `FUSED_ADDR`, plus reserved future forms like `WIDE_STORE`, `BITFIELD_EXTRACT`, `ZERO_EXTEND`) (`docs/tmp/jit-pass-framework-v5-design.md:393-581`) | **Partial** | The implementation has only four forms, and they are still named like v4 families: `ROTATE`, `WIDE_MEM`, `ADDR_CALC`, `COND_SELECT` (`vendor/linux-framework/include/uapi/linux/bpf.h:1500-1505`). The broader design vocabulary is missing: there is no `WIDE_STORE`, `BITFIELD_EXTRACT`, or `ZERO_EXTEND`, and the semantic names differ from the design's `WIDE_LOAD` / `FUSED_ADDR` (`docs/tmp/jit-pass-framework-v5-design.md:414-523`). For the four implemented forms, x86 native-choice mappings are correct in the current dispatcher (`vendor/linux-framework/kernel/bpf/jit_directives.c:523-542`; `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2499-2568`). |
| 5 | **Generic matcher**: one kernel matcher should work for all canonical forms (`docs/tmp/jit-pass-framework-v5-design.md:908-1155`) | **Partial** | The core matcher is generic: all v2 rules use `bpf_jit_match_pattern()` plus `bpf_jit_check_constraints()` through `bpf_jit_validate_pattern_rule()` (`vendor/linux-framework/kernel/bpf/jit_directives.c:1495-1635`). But the pipeline still has form-specific logic before lowering: `bpf_jit_pattern_rule_shape_valid()` hardcodes per-form allowed `site_len` values, and `bpf_jit_native_choice_valid()` hardcodes per-form native choices (`vendor/linux-framework/kernel/bpf/jit_directives.c:523-559`). So there is one generic matcher, but not a fully generic end-to-end rule contract. |
| 6 | **Parameterized emitters**: emitters consume canonical parameters extracted from bindings, not raw BPF instruction positions (`docs/tmp/jit-pass-framework-v5-design.md:547-585`, `docs/tmp/jit-pass-framework-v5-design.md:593-702`) | **Missing** | This is the main remaining design failure. `emit_bpf_cmov_select()` / `_compact()` still take raw instruction pointers (`vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:1353-1460`), and the wide/rotate/lea emitters still re-parse `insns[site_start + N]` and branch on `site_len` (`vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2111-2174`, `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2200-2331`, `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2348-2424`). The dispatcher passes raw instructions, not canonical params (`vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2482-2568`). |
| 7 | **Extensibility claim**: “new pattern variant = userspace-only change” (`docs/tmp/jit-pass-framework-v5-design.md:31-35`, `docs/tmp/jit-pass-framework-v5-design.md:1321-1473`) | **Partial** | Today this claim is true only for a narrow subset of variants: the new pattern must fit the exact-opcode DSL, an already-whitelisted `site_len`, and the emitter's existing positional assumptions. A new **7-insn rotate** cannot be userspace-only: the parser rejects it because rotate is whitelisted only for lengths 4/5/6 (`vendor/linux-framework/kernel/bpf/jit_directives.c:544-549`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1891-1900`), and the rotate emitter only knows how to decode 4/5/6-insn layouts (`vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2200-2277`). The lack of opcode masks also pushes userspace into descriptor explosion rather than true pattern abstraction: `COND_SELECT` currently enumerates 40 exact jump opcodes times move variants and shapes (`scanner/src/pattern_v5.cpp:340-359`). |
| 8 | **Security hardening**: overflow-safe bounds, kernel-owned CPU legality, sealed memfd + digest binding (`docs/tmp/jit-pass-framework-v5-design.md:706-779`) | **Partial (2.5 / 3)** | **Overflow-safe bounds** are now implemented: site end is computed via `check_add_overflow()` and reused by validators and lookup (`vendor/linux-framework/kernel/bpf/jit_directives.c:562-580`, `vendor/linux-framework/kernel/bpf/jit_directives.c:616-617`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1504-1505`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1646-1647`, `vendor/linux-framework/kernel/bpf/jit_directives.c:2124-2130`). **Sealed memfd + digest binding** are implemented: the kernel requires a sealed memfd and checks `insn_cnt` plus `prog_tag` against the loaded program (`vendor/linux-framework/kernel/bpf/jit_directives.c:31-40`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1987-2046`; `vendor/linux-framework/include/uapi/linux/bpf.h:1480-1490`). **Kernel-owned CPU legality** is only partially complete: the kernel now derives minimum required features from `(canonical_form, native_choice)` for `CMOVCC` and `RORX`, and x86 defensively rechecks BMI2 before emitting `rorx` (`vendor/linux-framework/kernel/bpf/jit_directives.c:583-592`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1652-1656`, `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2281-2285`), but the ABI still carries a user-provided `cpu_features_required` field and the kernel simply ORs it in rather than treating CPU legality as purely kernel-owned (`vendor/linux-framework/include/uapi/linux/bpf.h:1541-1542`, `vendor/linux-framework/include/uapi/linux/bpf.h:1589-1590`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1652-1654`). |
| 9 | **v4/v5 coexistence**: v1 (v4) and v2 (v5) blobs should coexist cleanly with a usable upgrade path (`docs/tmp/jit-pass-framework-v5-design.md:782-887`) | **Implemented** | Coexistence is clean at the ABI/parser level. The policy header exposes versions 1 and 2 (`vendor/linux-framework/include/uapi/linux/bpf.h:1471-1474`), `bpf_jit_parse_policy()` dispatches on `hdr->version` (`vendor/linux-framework/kernel/bpf/jit_directives.c:2066-2074`), and the internal `struct bpf_jit_rule` carries both legacy and v5 fields (`vendor/linux-framework/include/linux/bpf_jit_directives.h:54-81`). `bpf_jit_rule_form()` normalizes both paths onto the same x86 dispatcher (`vendor/linux-framework/kernel/bpf/jit_directives.c:504-521`; `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2427-2443`). The caveat is conceptual rather than mechanical: v2 currently reuses old positional emitters, so the upgrade path is operational but not yet fully design-conformant. |
| 10 | **Scanner library integration**: `scanner/` should be the single authoritative v5 implementation, and `kernel_runner.cpp` should become a thin wrapper (`docs/kernel-jit-optimization-plan.md:737`, `docs/tmp/v5-iteration-results.md:9-27`) | **Implemented, with one policy caveat** | This is now true for v5 scanning and blob serialization. `scanner/include/bpf_jit_scanner/pattern_v5.hpp` defines the shared v5 ABI mirror and API (`scanner/include/bpf_jit_scanner/pattern_v5.hpp:57-149`); `scanner/src/pattern_v5.cpp` contains the descriptor library, matcher, scanner, and blob builder (`scanner/src/pattern_v5.cpp:340-756`); the standalone scanner CLI uses the same functions (`scanner/src/cli.cpp:582-590`, `scanner/src/cli.cpp:625-633`); and `micro/runner/src/kernel_runner.cpp` now only calls `scan_v5_builtin()` and `build_policy_blob_v5()` (`micro/runner/src/kernel_runner.cpp:3`, `micro/runner/src/kernel_runner.cpp:767-816`). The remaining caveat is that the authoritative scanner still uses a greedy descriptor-order scan with `priority = 0` rather than the richer overlap/priority policy story (`scanner/src/pattern_v5.cpp:96-107`, `scanner/src/pattern_v5.cpp:653-685`). |

## 2. Overall Conformance Score

**6.5 / 10**

Why not lower:

- The earlier hardening gaps are mostly real fixes now: overflow-safe site arithmetic, CMOV on the v5 path, scanner unification, and kernel-derived minimum CPU legality all exist in code (`vendor/linux-framework/kernel/bpf/jit_directives.c:562-592`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1638-1657`; `scanner/src/pattern_v5.cpp:340-756`; `micro/runner/src/kernel_runner.cpp:767-816`).
- The rotate regression investigation also confirms that v4 and v5 currently converge to identical rotate/all code generation on the tested workload, so the new scanner path is not obviously changing the emitted machine code shape (`docs/tmp/rotate-v5-regression-investigation.md:19-24`, `docs/tmp/rotate-v5-regression-investigation.md:88-99`).

Why not higher:

- The design's central contract is still missing: the kernel does not extract canonical bindings and x86 emitters do not consume canonical parameters.
- The implemented pattern DSL is weaker than the design DSL.
- The current extensibility claim is only conditionally true.

## 3. What A Reviewer Will Challenge

- **“Where is the binding table you said made emitters independent of pattern shape?”** The design promises `binding_count` plus canonical-param extraction (`docs/tmp/jit-pass-framework-v5-design.md:593-702`, `docs/tmp/jit-pass-framework-v5-design.md:1149-1152`), but the actual ABI and internal rule structs have no bindings at all (`vendor/linux-framework/include/uapi/linux/bpf.h:1588-1600`; `vendor/linux-framework/include/linux/bpf_jit_directives.h:67-80`).
- **“Why are your ‘canonical forms’ still basically the old v4 families?”** The design's semantic vocabulary includes `WIDE_LOAD`, `FUSED_ADDR`, `WIDE_STORE`, `BITFIELD_EXTRACT`, and `ZERO_EXTEND` (`docs/tmp/jit-pass-framework-v5-design.md:414-523`), while the UAPI exposes only the four existing family aliases (`vendor/linux-framework/include/uapi/linux/bpf.h:1500-1505`).
- **“Is ‘new pattern = userspace-only’ actually true?”** Not for a 7-insn rotate variant: the kernel rejects that length before matching, and the rotate emitter only knows 4/5/6-insn layouts (`vendor/linux-framework/kernel/bpf/jit_directives.c:544-549`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1891-1900`; `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2200-2277`).
- **“Why does the implementation enumerate hundreds of exact cmov patterns instead of using the opcode-mask matcher from the design?”** Because the actual DSL is exact-opcode only. The scanner compensates by enumerating exact jump opcodes and move forms (`scanner/src/pattern_v5.cpp:340-359`), which is a direct mismatch with `opcode_mask` / `opcode_value` in the design (`docs/tmp/jit-pass-framework-v5-design.md:166-178`, `docs/tmp/jit-pass-framework-v5-design.md:929-930`).
- **“What are the real overlap semantics?”** The kernel sorts by `(site_start, -priority)` but lookup only returns a rule whose `site_start == insn_idx` and the scanner greedily skips ahead after the first descriptor-order match (`vendor/linux-framework/kernel/bpf/jit_directives.c:487-501`, `vendor/linux-framework/kernel/bpf/jit_directives.c:2109-2145`; `scanner/src/pattern_v5.cpp:653-685`).
- **“What is your subprogram story?”** The runner now scans the full translated program (`micro/runner/src/kernel_runner.cpp:759-764`), but the kernel still activates only rules wholly contained within a single subprogram (`vendor/linux-framework/kernel/bpf/jit_directives.c:1683-1718`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1944-1949`). That is safe, but it is weaker than a clean multi-function story.

## 4. Changes Needed To Close The Gaps

1. **Add the missing canonical-binding ABI.** Extend the v2 rule format with `binding_count` and a `struct bpf_jit_binding` matching the design (`docs/tmp/jit-pass-framework-v5-design.md:608-655`, `docs/tmp/jit-pass-framework-v5-design.md:823-839`). Production files: `vendor/linux-framework/include/uapi/linux/bpf.h`, `vendor/linux-framework/include/linux/bpf_jit_directives.h`, `vendor/linux-framework/kernel/bpf/jit_directives.c`, `scanner/include/bpf_jit_scanner/pattern_v5.hpp`, `scanner/src/pattern_v5.cpp`.
2. **Implement binding extraction in the kernel.** After `bpf_jit_match_pattern()` and `bpf_jit_check_constraints()`, add the missing `bpf_jit_extract_bindings()` stage from the design (`docs/tmp/jit-pass-framework-v5-design.md:1149-1152`) and store canonical params on the validated rule or on an emission-time scratch object.
3. **Refactor the x86 emitters to canonical-param interfaces.** Replace positional decoding in `emit_bpf_cmov_select()`, `emit_bpf_wide_load()`, `emit_bpf_rotate()`, and `emit_bpf_lea_fusion()` with `emit_canonical_*()` helpers that consume canonical params rather than `insns[site_start + N]` (`vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:1353-1460`, `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:2111-2424`). This is the change that actually makes pattern shape irrelevant to lowering.
4. **Upgrade the pattern ABI from exact opcode to masked opcode/value matching.** Add `opcode_mask` + `opcode_value` to the kernel and scanner mirror structs, update the kernel matcher and scanner matcher to use masked compare, and collapse the current `COND_SELECT` exact-opcode explosion into a small masked pattern family (`docs/tmp/jit-pass-framework-v5-design.md:166-178`, `docs/tmp/jit-pass-framework-v5-design.md:928-930`).
5. **Generalize or remove the hardcoded `site_len` whitelist.** `bpf_jit_pattern_rule_shape_valid()` should become a per-form maximum-span check, or better, an emitter-contract check based on extracted canonical params rather than raw site length (`vendor/linux-framework/kernel/bpf/jit_directives.c:544-559`). Without this, the “userspace-only” claim stays false for otherwise valid new variants.
6. **Align the canonical-form vocabulary with the design.** At minimum, rename or alias the current enum to the design's semantic names (`SELECT`, `WIDE_LOAD`, `ROTATE`, `FUSED_ADDR`) and reserve the additional forms called out in the design (`WIDE_STORE`, `BITFIELD_EXTRACT`, `ZERO_EXTEND`) even if their emitter slots stay `NULL` for now (`docs/tmp/jit-pass-framework-v5-design.md:393-581`).
7. **Make CPU legality fully kernel-owned.** Keep `bpf_jit_cpu_features_for_native_choice()` as the source of truth, stop depending on user-provided legality bits for implemented forms, and optionally reject blobs whose `cpu_features_required` disagrees with kernel-derived minimums instead of silently OR-ing them (`vendor/linux-framework/kernel/bpf/jit_directives.c:583-592`, `vendor/linux-framework/kernel/bpf/jit_directives.c:1652-1656`).
8. **Define real overlap semantics.** Either reject overlapping v5 rules outright in the parser, or implement interval-aware lookup plus non-greedy userspace arbitration so `priority` actually means something (`vendor/linux-framework/kernel/bpf/jit_directives.c:487-501`, `vendor/linux-framework/kernel/bpf/jit_directives.c:2109-2145`; `scanner/src/pattern_v5.cpp:96-107`, `scanner/src/pattern_v5.cpp:653-685`).
9. **Make the subprog story explicit and consistent.** Either keep the current single-subprog activation rule and state that v5 is intentionally single-subprog for now, or change userspace scanning / kernel activation together so multi-subprog support is a real property rather than a silent rejection path.

## 5. Work Estimate To Reach “Design-Conformant”

For the **current four-form prototype** (not counting full x86 implementations of future forms like `WIDE_STORE` / `BITFIELD_EXTRACT` / `ZERO_EXTEND`), the remaining work is roughly:

| Binding-table ABI + scanner mirror | 4-5 | 120-180 |

The six production files are:

## 6. Is The Current “Extensible Framework” Claim Defensible For A Paper?

**Not in the strong form currently written.**

What is defensible today:

- The kernel has a bounded declarative v2 matcher plus generic constraint interpreter.
- Userspace now owns a shared v5 pattern library and blob builder in `scanner/`.
- For some existing variants inside the current shape envelope, userspace can add descriptors without touching kernel code.

What is **not** defensible today:

- “New pattern variant = userspace-only change” as a general property.
- “Canonical lowering is decoupled from pattern shape.”
- “Canonical forms are the stable legality plane” in the full design sense.

The blocker is still the missing binding-table / parameterized-emitter layer. Until the kernel lowers from canonical parameters instead of raw BPF instruction positions, the implementation is best described as a **strong v5-front-end prototype over v4-style x86 lowering**, not as the fully design-conformant extensible framework described in `docs/tmp/jit-pass-framework-v5-design.md`.
