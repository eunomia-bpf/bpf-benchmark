# P81 Strict Feature/Complexity Reduction Review

Review target: review started on `main` at `6ab0083a`; during review P78 advanced `main`/`origin/main` to `6621d896`. Findings below cover the source surfaces visible across that old base, the P78 updates, and the remaining in-flight working-tree edit to `bpfopt/crates/bpfget/src/main.rs`. This was a read-only audit of source behavior; no build/test/KVM/AWS run was executed.

Core standard used here: if a surface is not on the benchmark path, not required by v3, and only helps debugging or compatibility, delete it. If it makes missing capability look like partial success, delete or make it fail.

## D. Total LOC Delta Estimate

Conservative cleanup: -1,300 to -1,800 LOC.

This assumes deleting public `scan-map-keys`, daemon `optimize-all`, profile save/load, status help shell-out, `disabled_passes`, `bpfrejit --dry-run`, `bpfprof --show`, text `bpfget --list`, config profiles/rules, fixed-point/debug-trace/pass helper leftovers, and tests that only defend those surfaces.

Aggressive cleanup: -4,000 to -5,200 LOC.

This adds deletion of the current `branch_flip` implementation, pruning `bpfprof` PMU/per-insn placeholders, simplifying `PassManager`/analysis/provider abstractions, shrinking daemon payload structs, moving map-key discovery out of `bpfopt`, and trimming redundant integration tests around the removed feature surfaces.

## E. Priority Ranking

1. Delete `disabled_passes` end to end. It is forbidden by the project rules, currently mostly empty/dead, and leaks into daemon, optimizer policy, runner provenance, and artifacts.
2. Delete public `bpfopt scan-map-keys` or move it behind a proper side-output boundary. It is the clearest v3 §10 violation and only exists as daemon plumbing.
3. Delete or quarantine current `branch_flip` + PMU profile surface. It is not in the default benchmark, does not have real per-site PGO input, and falls back to heuristics.
4. Slim daemon socket protocol: remove `optimize-all`, profile save/load, status help, and require `enabled_passes`. This aligns option B without touching runner Python.
5. Remove PassManager fixed-point/debug/unused API complexity. Low-risk reduction that makes the optimizer match the v3 single-pass pipeline model.

## F. 不该删的

- Do not delete the six CLI binaries as a boundary: `bpfopt`, `bpfget`, `bpfverify`, `bpfprof`, `bpfrejit`, `bpfrejit-daemon`. Current Cargo dependencies show each CLI depends on `kernel-sys`, not on another CLI crate, and direct BPF syscall usage is limited to `kernel-sys`.
- Do not delete `bpfopt optimize` or the individual pass subcommands for the paper-critical performance passes: `map_inline`, `const_prop`, `dce`, `skb_load_bytes_spec`, `bounds_check_merge`, `wide_mem`, `bulk_memory`, `rotate`, `cond_select`, `extract`, `endian_fusion`. They are the paper story and v3 CLI surface. The exception is current `branch_flip`: keep only if it becomes real PGO-only.
- Do not delete `bpfget --full` or `bpfget --target`. They are the daemon's side-input source for bytecode, program metadata, map FD replay, BTF metadata, and kinsn target JSON.
- Do not delete `bpfverify --report`, `--verifier-states-out`, `--map-fds`, `--fd-array`, `--func-info`, or `--line-info`. These are not debug verbosity; they preserve load context, verifier-state side inputs, and BTF replay.
- Do not delete `bpfrejit --fd-array` or `--map-fds`. They are required to submit replacement bytecode with kinsn and map references intact.
- Do not delete daemon map invalidation tracking while `map_inline` exists. Dynamic map inlining is core to the paper, and v3 still gives the daemon watch/invalidation responsibility.
- Do not refactor runner Python away from the daemon socket boundary during v3 migration. The allowed runner changes are bug fixes and stale data updates, not a new CLI pipeline orchestration layer.
- Do not delete core structured pass result fields: `pass`, `changed`, `sites_applied`, `insn_delta`, `verify_result`, `verify_error`, and `action`. Delete duplicates and debug shells, not the v3 audit record itself.
- Do not delete `bpfopt analyze` or `list-passes` just because they are not raw-bytecode transforms. v3 explicitly names them as tool subcommands. The stricter fix is to keep them out of transform pipelines and avoid daemon dependence on `list-passes` for `status`.
