# Kernel-Fixed Baselines Experiment Design

**Document purpose:** TODO #21 / TODO #44 — Design the "kernel-fixed baselines" experiment, the OSDI go/no-go condition.

**Date:** 2026-03-10

**Status:** Design only — no code changes.

---

## 1. Motivation and Framing

The mandatory falsification condition (from `docs/kernel-jit-optimization-plan.md` §5.4):

> If fixed kernel policies achieve the same gains in all tested hardware and workload configurations, the correct conclusion is "use kernel peepholes," not "publish a userspace-guided interface."

The experiment tests whether the v4 framework's results can be replicated by simply hardcoding the same pattern-match logic unconditionally inside `do_jit()` — with no policy blob, no `BPF_PROG_JIT_RECOMPILE` syscall, and no userspace involvement. This is the strongest possible "why not just" challenge from a reviewer:

> "Why not submit three peephole patches to the kernel? Why does userspace need to be involved at all?"

The answer must be empirical, not merely theoretical. We need data showing:
1. At least one optimization is **policy-sensitive** — the fixed heuristic gives the wrong answer on some workload/CPU combination.
2. The framework's extensibility can be demonstrated quantitatively (LOC to add a new rule).
3. Where the fixed heuristic happens to match, we must say so honestly.

## 2. The Three Fixed Heuristics

### 2.1 Fixed ROTATE Heuristic

**Pattern:** Any consecutive BPF instruction sequence that satisfies one of:
- `bpf_jit_validate_rotate_4insn()` (canonical 4-insn: mov + lsh + rsh + or, or commuted)
- `bpf_jit_validate_rotate_5insn()` (5-insn two-copy variant)
- `bpf_jit_validate_rotate_5insn_masked()` (5-insn masked variant)
- `bpf_jit_validate_rotate_6insn()` (6-insn masked 32-bit variant)

**Fixed action:** Always emit `ROR r, imm8` (or `RORX r, r, imm8` if BMI2 is available at boot time).

**Expected coverage:** 126 sites in current test programs (115 in rotate64_hash, 11 in packet_rss_hash). Coverage will track exactly the v4 framework's ROTATE coverage.

**Profitability question:** Is rotate idiom recognition always beneficial? The fixed heuristic assumes yes. This seems reasonable — rotating by a constant is mathematically equivalent regardless of workload or CPU. The only CPU-gating issue is RORX (BMI2), which the fixed heuristic can also gate with `boot_cpu_has(X86_FEATURE_BMI2)`.

**Honest assessment for paper:** The fixed ROTATE heuristic will almost certainly match the framework's results. Rotate idiom recognition is a textbook peephole with no policy-sensitivity — the right answer does not depend on branch predictability, workload data distribution, or operator preference. This is a point to concede honestly in the paper.

### 2.2 Fixed WIDE_MEM Heuristic

**Pattern:** Consecutive BPF byte-load sequences matching either:
- `bpf_jit_validate_wide_mem_low_first()` (low-byte-first N=2,4,8)
- `bpf_jit_validate_wide_mem_high_first()` (high-byte-first 2-byte, clang output)

**Fixed action:** Always emit a single `MOVZX` / `MOV` wide load (2-byte → `movzx r16`, 4-byte → `mov r32`, 8-byte → `mov r64`).

**Expected coverage:** 3 programs, 11 sites (load_byte_recompose: 1 site, stride_load_4: 2 sites, additional programs in full suite).

**Profitability question:** Is merging byte-loads always beneficial? Generally yes — wider accesses have better cache behavior and fewer instruction-fetch pressure. However there are edge cases:
- If the bytes come from non-contiguous memory (which the validator rules out by checking consistent base+offset), the wide load is unsafe. The validator already handles correctness.
- On highly cache-resident data, the difference between 1-byte vs 2-byte loads may be negligible. But this is noise, not a systematic reversal.

**Honest assessment for paper:** The fixed WIDE_MEM heuristic will match the framework's results on current test programs. However, the framework's richer pattern vocabulary (the distinction between high-first and low-first variants had to be discovered by debugging clang output over three rounds) demonstrates that the set of patterns is open-ended. A fixed kernel heuristic that ships with kernel 7.2 cannot recognize patterns generated by clang 19 that clang 17 didn't produce. The framework's extensibility argument is strongest here — the pattern evolution history (Round 1 → Round 2 scanner fixes) shows that patterns change as compilers evolve.

### 2.3 Fixed LEA Heuristic

**Pattern:** `bpf_jit_validate_addr_calc_rule()`:
```
[0] mov64  dst, src
[1] lsh64  dst, K    (K in {1, 2, 3})
[2] add64  dst, base
```

**Fixed action:** Always emit `lea dst, [src * (1<<K) + base]`.

**Expected coverage:** 4-5 programs, 5 sites in current test programs (stride_load_16, multi_acc_4, etc.).

**Profitability question:** LEA fusion is a classic backend peephole with uniformly positive benefit. On all modern x86 microarchitectures tested (Skylake, Zen), LEA with scale 2/4/8 is lower latency than the three-instruction sequence. There is no workload or CPU scenario where the three-instruction sequence is preferable.

**Honest assessment for paper:** The fixed LEA heuristic will match the framework exactly. This is the weakest case for the framework. Concede this clearly.

## 3. Why CMOV is Different (The Key Policy-Sensitive Case)

The fixed CMOV heuristic would be:

> Always replace the 4-insn diamond (cond-jmp + mov_false + ja + mov_true) or 3-insn compact (mov_default + cond-jmp + mov_override) with `cmovcc`.

**Why this is wrong:** The existing data shows:
- `log2_fold`: baseline 764ns → cmov 824ns (+7.9%). Predictable branches are faster as branches.
- `switch_dispatch`: no-cmov ablation +26.3% slower. Random branches benefit from cmov.
- `binary_search`: no-cmov ablation +11.7% slower.
- `bounds_ladder`: no-cmov ablation -18.1% (cmov *hurts*).
- `large_mixed_500`: no-cmov ablation -23.9% (cmov *hurts*).

A fixed "always cmov" heuristic would:
- Help programs with unpredictable branches
- Hurt programs with highly predictable branches
- Be systematically wrong for the predictable branch case

This is the paper's core claim: **the profitability of cmov depends on runtime workload characteristics that are not visible at JIT-compile time**. Specifically, branch predictability is a function of the data being processed, not the instruction sequence itself.

The v4 framework allows userspace to measure branch behavior (via perf counters or profiling) and apply cmov only where it helps. A fixed kernel heuristic cannot do this.

**cmov is the go/no-go experiment.** If we can show:
- Fixed ROTATE/WIDE/LEA ≈ framework (honest concession)
- Fixed CMOV ≠ framework on predictable-branch workloads (policy-sensitive)

Then the paper's argument is: the framework is needed *at minimum* for CMOV. The framework's extensibility makes it the right substrate for all four families.

## 4. Implementation Approach

### 4.1 Where to Add Fixed Heuristics in `do_jit()`

The v4 policy dispatch block in `do_jit()` is at approximately line 2598:

```c
/* v4 rule dispatch: check if a rewrite rule covers this insn */
{
    const struct bpf_jit_policy *v4_policy;
    ...
    if (v4_policy && v4_policy->active_cnt) {
        ...
        consumed = bpf_jit_try_emit_rule(...);
        if (consumed > 0) { ... continue; }
    }
}
```

The fixed heuristics should be added **after** the v4 policy dispatch block, so the policy always wins when present. Fixed heuristics fire only when no policy rule covers the instruction.

### 4.2 Implementation Sketch: Fixed ROTATE

```c
/* Fixed ROTATE heuristic (after v4 dispatch block) */
if (IS_ENABLED(CONFIG_X86_BPF_JIT_FIXED_ROTATE)) {
    int site_len = 0;
    if (bpf_jit_validate_rotate_4insn(bpf_prog->insnsi, bpf_prog->len, i - 1))
        site_len = 4;
    else if (bpf_jit_validate_rotate_5insn(bpf_prog->insnsi, bpf_prog->len, i - 1))
        site_len = 5;
    else if (bpf_jit_validate_rotate_5insn_masked(bpf_prog->insnsi, bpf_prog->len, i - 1))
        site_len = 5;
    else if (bpf_jit_validate_rotate_6insn(bpf_prog->insnsi, bpf_prog->len, i - 1))
        site_len = 6;

    if (site_len > 0 &&
        !bpf_jit_site_has_side_effects(bpf_prog->insnsi, i - 1, site_len)) {
        /* Create a synthetic rule for the emitter */
        struct bpf_jit_rule fixed_rule = {
            .rule_kind     = BPF_JIT_RK_ROTATE,
            .site_start    = i - 1,
            .site_len      = site_len,
            .native_choice = boot_cpu_has(X86_FEATURE_BMI2) ?
                             BPF_JIT_ROT_RORX : BPF_JIT_ROT_ROR,
        };
        int consumed = bpf_jit_try_emit_rule(&prog, bpf_prog,
                                              &fixed_rule,
                                              bpf_prog->insnsi,
                                              priv_frame_ptr != NULL);
        if (consumed > 0) {
            /* Same addrs[] bookkeeping as the v4 path */
            ...
            continue;
        }
    }
}
```

Key insight: the existing `bpf_jit_try_emit_rule()` dispatcher and all emitters (`emit_bpf_rotate()`, etc.) can be reused **unchanged**. The fixed heuristic just creates a synthetic rule on the stack and calls the same emitter.

### 4.3 Implementation Sketch: Fixed WIDE_MEM

```c
/* Fixed WIDE_MEM heuristic */
if (IS_ENABLED(CONFIG_X86_BPF_JIT_FIXED_WIDE_MEM)) {
    /* Try each supported site_len: 2-byte (len=4 high-first, len=1 low-first),
     * 4-byte (len=10), 8-byte (len=22) */
    static const u32 try_lens[] = {4, 10, 22};
    int k;
    for (k = 0; k < ARRAY_SIZE(try_lens); k++) {
        u32 site_len = try_lens[k];
        struct bpf_jit_rule fixed_rule = {
            .rule_kind     = BPF_JIT_RK_WIDE_MEM,
            .site_start    = i - 1,
            .site_len      = site_len,
            .native_choice = BPF_JIT_WMEM_WIDE_LOAD,
        };
        if (bpf_jit_validate_wide_mem_rule(bpf_prog->insnsi,
                                           bpf_prog->len,
                                           &fixed_rule)) {
            int consumed = bpf_jit_try_emit_rule(&prog, bpf_prog,
                                                  &fixed_rule, ...);
            if (consumed > 0) { ... continue; }
        }
    }
}
```

Note: `bpf_jit_validate_wide_mem_rule()` already accepts a `struct bpf_jit_rule *` and returns bool, so it can be called with a synthetic rule without any modification.

### 4.4 Implementation Sketch: Fixed LEA

```c
/* Fixed LEA heuristic */
if (IS_ENABLED(CONFIG_X86_BPF_JIT_FIXED_LEA)) {
    struct bpf_jit_rule fixed_rule = {
        .rule_kind     = BPF_JIT_RK_ADDR_CALC,
        .site_start    = i - 1,
        .site_len      = 3,
        .native_choice = BPF_JIT_ACALC_LEA,
    };
    if (bpf_jit_validate_addr_calc_rule(bpf_prog->insnsi,
                                        bpf_prog->len,
                                        &fixed_rule)) {
        int consumed = bpf_jit_try_emit_rule(&prog, bpf_prog,
                                              &fixed_rule, ...);
        if (consumed > 0) { ... continue; }
    }
}
```

### 4.5 Implementation Sketch: Fixed CMOV (the control condition)

```c
/* Fixed CMOV heuristic — ALWAYS apply, regardless of branch predictability */
if (IS_ENABLED(CONFIG_X86_BPF_JIT_FIXED_CMOV)) {
    struct bpf_jit_rule fixed_rule = {
        .rule_kind     = BPF_JIT_RK_COND_SELECT,
        .native_choice = BPF_JIT_SEL_CMOVCC,
        /* site_len and site_start: determined by matching diamond/compact */
    };
    /* ... match diamond (len=4) or compact (len=3) ... */
    /* Call bpf_jit_validate_cond_select_rule(), then try_emit_rule() */
}
```

The fixed CMOV heuristic is the **control experiment**: it shows what happens when you apply cmov without workload knowledge.

### 4.6 Kconfig Flags

Each fixed heuristic should be a separate Kconfig option:
```
CONFIG_X86_BPF_JIT_FIXED_ROTATE  (tristate, default n)
CONFIG_X86_BPF_JIT_FIXED_WIDE_MEM (tristate, default n)
CONFIG_X86_BPF_JIT_FIXED_LEA     (tristate, default n)
CONFIG_X86_BPF_JIT_FIXED_CMOV    (tristate, default n)
```

This lets us build one kernel image per heuristic, avoiding interaction effects, and makes it easy to ablate each optimization independently.

Alternatively: a single runtime flag via `bpf_jit_enable` sysctl extension or a new sysctl `bpf_jit_fixed_heuristics` bitmask. The Kconfig approach is simpler for paper experiments.

### 4.7 Key Property: No New Kernel Infrastructure

The fixed heuristics reuse:
- All existing validators (`bpf_jit_validate_*`) from `jit_directives.c`
- All existing emitters (`emit_bpf_rotate`, `emit_bpf_wide_load`, `emit_bpf_lea_fusion`, `emit_bpf_cmov_select`) from `bpf_jit_comp.c`
- The same `bpf_jit_try_emit_rule()` dispatch function

Only new code: 4 × ~20 lines in `do_jit()` to match patterns and construct synthetic rules. Total: ~80 LOC.

This is important: it shows that the existing validators + emitters are not "framework-only" code — they can be used by a fixed heuristic too. The difference is who calls them and when.

### 5.3 Measurement Protocol

For each (program × variant) pair:
- 30 iterations × 1000 repeats (matching rigorous rerun protocol)
- CPU pinned to isolated core
- Performance governor enabled
- Turbo disabled (for reproducibility)
- Metrics: `exec_ns` (primary), `jited_prog_len` (code size), `xlated_prog_len` (invariant check)

### 5.4 Key Comparisons

**Comparison A: Framework parity for substrate directives**

```
For each of {ROTATE, WIDE_MEM, LEA}:
  H0: kernel-fixed-X exec_ns == kernel-v4-policy exec_ns
  Expected: ACCEPT H0 (fixed ≈ framework)
```

If this holds, we concede these three families in the paper.

**Comparison B: CMOV policy-sensitivity**

```
Programs: log2_fold (predictable branches), switch_dispatch (random branches)
Compare:  kernel-fixed-cmov vs kernel-v4-policy (CMOV disabled for log2_fold)
Expected:
  log2_fold:      fixed-cmov +~8% slower than no-cmov policy
  switch_dispatch: fixed-cmov faster than no-cmov policy
```

If this holds, it proves that no single fixed heuristic gets the right answer on all programs.

**Comparison B2: Framework-smart vs Fixed-ALL (the killer argument)**

```
All programs with both ROTATE and CMOV sites:
Compare: kernel-fixed-all vs kernel-v4-smart

For log2_fold:
  fixed-all: ROTATE ✅ + CMOV forced → net HURT (+8%)
  v4-smart:  ROTATE ✅ + CMOV skipped (profile says predictable) → net HELP

For rotate64_hash:
  fixed-all: ROTATE ✅ → -28%
  v4-smart:  ROTATE ✅ → -28% (same)

Key point: fixed-all is all-or-nothing. Framework selectively applies.
```

**Comparison C: Combined speedup**

```
kernel-fixed-all vs kernel-v4-policy vs kernel-stock
For rotate64_hash: -28% target
For packet_rss_hash: -10% target
```

**Comparison D: Extensibility (not a performance metric)**

```
LOC to add new fixed heuristic: ~80 LOC in do_jit() + required kernel release cycle
LOC to add new v4 framework rule: ~50 LOC validator + ~50 LOC emitter + 0 kernel release cycle (userspace only for new patterns in v5 design)
```

### 6.3 Honest Assessment for the Paper

The paper should present this experiment with intellectual honesty:

**Where the framework adds nothing over fixed heuristics:**
- ROTATE: fixed heuristic matches exactly, because rotate idioms are always beneficial
- WIDE_MEM: fixed heuristic matches for current patterns, but pattern evolution requires framework extensibility
- LEA: fixed heuristic matches exactly, because LEA is universally beneficial on x86

**Where the framework is essential:**
- CMOV: profitability is workload-dependent. Fixed "always cmov" hurts predictable-branch programs by +7.9%. The framework lets userspace profile and apply cmov only where it helps.

**The extensibility argument (cannot be falsified by performance comparison):**
- Adding ROTATE patterns to the kernel took 3 rounds of iteration (Round 1: 0 coverage, Round 2: 0 coverage, Round 3: 126 sites after extending the validator for commuted/5-insn/masked variants). This iteration happened in userspace (scanner pattern changes) + kernel (new validator sub-cases). In the v5 design, only the userspace scanner would need updating — no kernel change.
- This demonstrates that the pattern space is not closed. Future compiler output changes (new clang version, new BPF backend optimizations) will produce new patterns that cannot be anticipated in a fixed heuristic.

## 7. Paper Framing

### 7.1 Section Placement

The kernel-fixed baselines experiment should go in the evaluation section as a dedicated subsection:

> **§7.4 Comparison with Fixed Kernel Heuristics**
>
> To understand which optimizations require userspace policy and which could be safely hardcoded, we implement fixed heuristics for all four directive families and compare against the framework with an explicit policy.

### 7.2 Key Claims to Make

1. **For substrate directives (ROTATE, WIDE, LEA):** The framework achieves the same performance as a fixed heuristic. We do not claim the framework is necessary for these families — a kernel patch would achieve the same result. The framework's value here is extensibility and deployment control.

2. **For CMOV:** The fixed heuristic is *wrong* on predictable-branch programs. The framework correctly handles this by allowing userspace to disable cmov for such programs. This is the go/no-go condition.

3. **Why not just submit three patches?** Because CMOV cannot be hardcoded. And because the framework provides the substrate for all future optimizations (including ones we haven't discovered yet) without kernel changes.

### 7.3 Reviewer Challenge and Response

**Likely reviewer challenge:**
> "For rotate and wide loads, the fixed heuristic works just as well. You only need the framework for CMOV. Why not submit a cmov heuristic to the kernel that checks branch history at JIT time via BTB probing or similar?"

**Response (to pre-populate in paper):**
1. JIT time ≠ runtime. Branch history at first-load JIT time reflects the first workload, not subsequent ones. A server that processes both predictable (audit) and random (security scan) traffic cannot be optimized at JIT time.
2. Fleet A/B testing: even if you had a good heuristic, you cannot run it on 5% of machines first with a fixed kernel heuristic. The framework makes this possible.
3. Service ownership: Cilium's workload engineers should own Cilium's JIT policy, not Linux kernel maintainers who have no visibility into Cilium's traffic patterns.
4. Future directives: branch_reorder (verifier-level CFG permutation) has the same workload-sensitivity as CMOV. The framework provides the substrate for these future optimizations.

## 8. Implementation Plan

### 8.1 Kernel Changes (Branch: `jit-fixed-baselines`)

**File: `vendor/linux/arch/x86/net/bpf_jit_comp.c`**

Location: Inside `do_jit()`, after the existing v4 policy dispatch block (approximately line 2655), before the main BPF-to-x86 switch statement.

Add four guarded blocks, one per heuristic. Each block:
1. Calls the existing validator function
2. Constructs a synthetic `struct bpf_jit_rule` on the stack
3. Calls `bpf_jit_try_emit_rule()`
4. Does the same `addrs[]` bookkeeping as the v4 path
5. `continue`s the BPF instruction loop if consumed > 0

The guards can be:
- `CONFIG_X86_BPF_JIT_FIXED_*` Kconfig options (recommended for paper experiments)
- Or a runtime bitmask sysctl (simpler deployment)

**File: `vendor/linux/kernel/bpf/Kconfig` (or `arch/x86/net/Kconfig`)**

Add four bool options, default n.

**No changes to:** `jit_directives.c`, any validator, any emitter, any UAPI. This is purely additive to `do_jit()`.

### 8.2 Userspace / Benchmark Changes

No new benchmark programs needed. The same test programs used in Round 3 cover all four families.

New runner flag: `--fixed-kernel-heuristic` (or a separate kernel image per variant). Recommend separate kernel images for clarity.

New benchmark YAML entries: add variants for each fixed-kernel kernel image to `config/micro_pure_jit.yaml` or a new `config/micro_fixed_baselines.yaml`.

### 8.3 Analysis Script

A new `micro/analyze_fixed_baselines.py` that:
1. Loads results for (stock, fixed-rotate, fixed-wide, fixed-lea, fixed-cmov, fixed-all, v4-policy)
2. Computes (fixed - v4-policy) / v4-policy for each program × directive
3. Outputs a table: "framework overhead" (should be ~0 for substrate, significant for CMOV)
4. Generates Figure: 3-panel bar chart showing exec speedup for stock vs fixed vs framework

## 10. Open Questions

1. **Are the validate_* functions currently `static`?** If so, they need to be exposed from `jit_directives.c`. Check the header `include/linux/bpf_jit_directives.h` — if they're declared there, they're already accessible. If not, add declarations.

2. **Interaction between fixed heuristics:** If `fixed-cmov` and `fixed-rotate` are both enabled, and a program has both patterns, can they interfere? The dispatch order (v4 first, then rotate, then wide, then lea, then cmov) should handle this, but needs verification.

3. **addrs[] bookkeeping for multi-insn sites:** The v4 dispatch path at line 2642 shows the correct pattern:
   ```c
   for (j = 0; j < consumed - 1; j++)
       addrs[i + j] = region_start;
   addrs[i + consumed - 1] = proglen;
   ```
   The fixed heuristics must use the same bookkeeping. This is important for JIT convergence correctness.

4. **cmov_select subprog issue:** The current v4 scanner finds 0 cmov sites in `cmov_select.bpf.c` due to the subprog boundary limitation (the diamond patterns are in a called subprog, and the scanner only scans the main prog's xlated range). The fixed heuristic scans all `bpf_prog->insnsi` during JIT, which may naturally cover subprogs (since libbpf inlines them). This could mean the fixed heuristic has **higher coverage** than the v4 framework for cmov, making the comparison stronger.

5. **For paper submission timing:** This experiment should be done before drafting §7.4. The analysis needs actual numbers, not predictions.

## 11. Summary: Framework vs Fixed Heuristic

| Directive | Fixed heuristic matches? | Why framework still needed |
|-----------|:---:|--------------------------|
| ROTATE | **Yes** | Extensibility: new clang patterns (e.g., 6-insn masked) would require kernel patch. In v5 design, only userspace scanner update needed. |
| WIDE_MEM | **Yes** | Same extensibility argument + high-byte-first pattern was discovered through debugging (not anticipated in advance). |
| LEA | **Yes** | No policy-sensitivity argument, but framework provides uniformity of mechanism. |
| CMOV | **No** | Profitability is workload-dependent. Fixed heuristic hurts predictable-branch programs by +7.9%. Framework allows per-program policy. |

The experiment's honest conclusion for the paper:

> Three of four optimizations could be implemented as fixed kernel heuristics with identical performance. The fourth (cmov_select) cannot: the correct lowering depends on runtime workload characteristics. We implement and measure all four fixed heuristics to provide this honest baseline. We conclude that the framework is necessary for policy-sensitive optimizations (cmov), provides deployment control that fixed heuristics cannot (fleet A/B testing, service-owned policy), and offers extensibility that avoids repeated kernel patch cycles as compiler output evolves.
