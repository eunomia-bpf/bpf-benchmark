## Executive conclusion

This is not just the known Tracee `syscall__init_module` `BPF_PROG_REJIT:
EINVAL` path. The latest KVM corpus artifact has 446 unique
`error_programs`, while the 2026-04-28 AWS x86 report had only the accepted
Tracee `EINVAL` in corpus and only Tracee failing in E2E.

The failures split into two different classes:

1. Definite daemon/bpfverify load-context replay gaps before optimization.
   `ORIG_VERIFY`, `Tracing programs must provide btf_id`, and
   `fd_idx without fd_array is invalid` occur while the daemon asks
   `bpfverify --verifier-states-out` to verify the original `prog.bin`.
   These are not pass-generated bytecode bugs; they mean the standalone
   verifier invocation is missing context that the original app loader had.
2. Final `opt.bin` dry-run rejections. `EINVAL` and `EACCES` dominate the
   remaining failures, but the stored artifact usually contains only errno,
   not full verifier text. These may be transform bugs or the same context
   replay problem on optimized bytecode. Based on the commit under test and
   the affected shapes, the highest pass suspicion is the
   `map_inline` + `const_prop` + `dce` chain, but the current artifact is not
   enough to prove a single pass.

Immediate broad rollback of all passes is not justified by the saved evidence.
The next useful action is to preserve full daemon workdirs/logs and rerun a
small verbose reproduction, then bisect pass lists. If an immediate green run is
needed before that, disable `map_inline` or the `map_inline,const_prop,dce`
cluster as a diagnostic rollback, not as permanent program filtering.

## Daemon log and verifier-log preservation

The daemon does have log locations during the run:

- `runner/libs/rejit.py` starts the daemon with `daemon.stdout.log` and
  `daemon.stderr.log` under a temporary socket directory.
- That socket directory is deleted on normal stop.
- `daemon/src/commands.rs` creates per-program workdirs named
  `bpfrejit-daemon-optimize-<pid>-<id>` under `std::env::temp_dir()`.
  Those contain `prog.bin`, `opt.bin`, `verified.bin`, `map_fds.json`,
  `fd_array.json`, `verifier_states.json`, and `bpfopt_report.json`.
- `WorkDir::drop` removes those per-program directories.

The saved KVM artifacts only contain `result.json`, `result.md`,
`progress.json`, and `metadata.json`; no daemon stdout/stderr logs or
per-program workdirs were preserved. A host search found no current
`daemon.stdout.log`, `daemon.stderr.log`, `verifier_states.json`, or
`bpfopt_report.json` from this run.

`bpfverify` behavior also explains the missing verifier text:

- `--report` or `--verifier-states-out` forces verifier log level 2.
- The daemon uses `--verifier-states-out` on the original program when
  `const_prop` is requested, so these original-verify failures often include
  verifier snippets.
- The daemon's final `opt.bin` dry-run does not request `--report` or a log
  level, so final `EINVAL`/`EACCES` usually records only errno.
- `bpfrejit` syscall failures preserve only a truncated verifier log summary.

Therefore a definitive final verifier-message classification requires a rerun
with verbose/preserved temp mode. Specifically, preserve
`/var/tmp/bpfrejit-runtime/run.x86-kvm.*`, daemon stdout/stderr, and
`bpfopt_report.json`; also make final `bpfverify` failures emit a full report.

- E2E had Tracee `error` for known `EINVAL` in cycle 0/1; `tetragon`,
  `bpftrace`, `bcc`, and `katran` were `ok`.

Current KVM x86 at `70e2fb26` differs materially:

- Corpus has 11 app runs in error and 446 unique saved error programs.
- E2E has all five cases in error.
- Tracee still contains the known `syscall__init_module`-class `EINVAL`, but

This makes "known Tracee kernel limitation only" an insufficient explanation.
It also argues against permanent app/type filtering: the failures should
surface as errors, and the root cause should be fixed at replay context,
logging, or pass correctness.

- `a1b8bade1 bpf: refactor register usage detection for BPF programs`
  touched `arch/x86/net/bpf_jit_comp.c`.
- `0725c44fd Relax REJIT poke table insn index check`
- `b07468207 bpf: fix verifier patch buffer restore` touched
  `include/linux/bpf_verifier.h` and `kernel/bpf/verifier.c`.
- `81608ed9e bpf: fix 4 review issues...` touched verifier/JIT-adjacent code.
- `9be94921f bpf: Add native emit check for kinsn and improve verifier logic`
  changed `kernel/bpf/syscall.c`.

So the fork verifier/JIT stack is not stable. However, the latest artifact
still points first at user-space replay/context and pass-side evidence gaps:
many failures occur in `bpfverify --verifier-states-out` on original bytecode,
and final dry-run failures lack the verifier text needed to prove a kernel-side
strictness change.

## Pass suspicion analysis

No single pass can be proven from the compact artifacts because successful and
failed per-pass `bpfopt_report.json` files were deleted. The following ranking
is based on failure stage, affected program shapes, and the commit under test.

| suspect | likelihood | rationale |
| --- | --- | --- |
| Daemon/bpfverify load-context replay | Very high | `ORIG_VERIFY`, `MISSING_BTF_ID`, and `FD_ARRAY` happen before `bpfopt optimize`. The daemon currently replays `--prog-type`, `--map-fds`, and sometimes `--fd-array`, but tracing/LSM attach metadata and some fd-array needs are missing in saved failures. |
| `map_inline` + `const_prop` + `dce` chain | High | The default pass list includes all three; commit `5fbe934b` changed `map_inline`, scan-map-key handling, and daemon support for live map values. Many preserved snippets involve map lookups/map values/null checks, and `const_prop` is why original verifier states are requested. A wrong map-inline constant or removed null/path check would naturally produce final verifier rejection. |
| `wide_mem` | Medium-low | Could explain real memory-access verifier errors, especially for XDP/TC, but the saved dominant failures are kprobe/tracing and original-verification/context failures. No preserved final log proves wide memory OOB. |
| `skb_load_bytes_spec` | Medium-low | Relevant to `sched_cls` programs such as Calico/Katran, but the saved Calico/Katran XDP/TC evidence is mostly original-verify failure or errno-only final `EACCES`. |
| `bounds_check_merge` | Low-medium | Could be unsafe if it weakens verifier-recognized bounds, but no preserved verifier message shows an OOB access or bad branch bound. |
| `bulk_memory`, `rotate`, `cond_select`, `extract`, `endian_fusion` | Low from current artifacts | No saved failure text points to illegal encoding, OOB jump, or register-type mismatch uniquely associated with these passes. |

Important nuance: a program with `changed=false` in an error record can still
fail before pass execution because the daemon must collect side inputs first.
Those failures should not be counted as pass regressions.

## Recommended fix direction

1. Preserve evidence on failure before changing pass logic.
   Keep daemon stdout/stderr and per-program `bpfrejit-daemon-optimize-*`
   directories for failed programs, at least in verbose/debug mode. The needed
   files are `prog.bin`, `opt.bin`, `verified.bin`, `map_fds.json`,
   `fd_array.json`, `target.json`, `verifier_states.json`,
   `map_values.json`, `bpfopt_report.json`, and `bpfrejit_summary.json`.

2. Make final verifier failures log-rich.
   Run final `bpfverify` with a report/log on failure so `EINVAL` and `EACCES`
   classify into actual verifier messages: illegal instruction, register type
   mismatch, invalid mem access, bad jump, missing attach metadata, etc.

3. Fix load-context replay before blaming transforms.
   The daemon/CLI path needs to replay enough load metadata for original and
   optimized verification:
   `btf_id`/attach metadata for tracing and LSM programs, expected attach type,
   and fd-array side input whenever the bytecode requires fd indexes or kinsn
   calls. The BCC tracing and Cilium failures are concrete examples.

4. After logs are preserved, bisect passes on a small failing set.
   Do not run full `vm-corpus` for the first pass. Use a few failing program
   workdirs or a small app reproduction and compare pass lists:
   `map_inline` only, `const_prop` only, `map_inline,const_prop`,
   `map_inline,const_prop,dce`, then add `wide_mem` and packet-specific passes.

5. Do not add permanent app/program-type exclusions.
   Any temporary pass disable should be a diagnostic rollback of a pass or pass
   cluster, not a ReJIT program filter. This keeps the framework aligned with
   the no-filtering design rule.

## Rollback or sanity-check recommendation

- No broad rollback of all 11 default passes based on the current artifact.
- If a green smoke is required before verbose reproduction, the least
  speculative temporary rollback is `map_inline` first, or
  `map_inline,const_prop,dce` as a cluster, because that is the changed area in
  `5fbe934b` and it matches map-value/null-check-heavy snippets.
- Add stricter sanity checks before `BPF_PROG_REJIT`: final dry-run with full
  replay context and full log capture. On failure, record the verifier report
  and exit with the error; do not silently skip or filter programs.

## Need rerun with verbose mode

Yes. The existing daemon log and per-program verifier artifacts are missing.
The current report can classify failure stage and errno, and can identify
definite context gaps, but it cannot answer the detailed `EINVAL`/`EACCES`
verifier messages for most final dry-run failures. A focused verbose rerun is
required to prove whether the final rejections are map-inline/const-prop bugs,
missing load context, kernel verifier changes, or a combination.
