# Paper Review R2: `paper.tex`

Date: 2026-03-10

Score: 6.0 / 10

The paper has a strong systems story, and most of the rigorous re-JIT microbenchmark numbers are internally consistent with the latest source artifacts. In particular, the per-family medians, CIs, and p-values in `paper.tex:980-984` match `docs/tmp/rigorous-benchmark-results.md:28-106`, and the paper is appropriately honest that WIDE_MEM is not yet a significant latency win. The main remaining problems are not the core thesis; they are stale or mislabeled numbers, and production-corpus framing that will look selective to a reviewer.

## Findings

1. **[CRITICAL] The runtime-suite numbers are stale and conflict with the current authoritative characterization data.** `paper.tex:204`, `paper.tex:321`, and `paper.tex:346` say the paper covers `11` runtime microbenchmarks and reports a runtime geomean of `0.875x`. The current characterization source says the authoritative runtime run is `9` cases with geomean `0.829x` and `7/9` significant: `docs/micro-bench-status.md:6`, `docs/micro-bench-status.md:433`, `docs/micro-bench-status.md:491`; `docs/paper-comparison.md:171`, `docs/paper-comparison.md:597`. This is a hard factual mismatch, not just a wording issue. Fix: update all runtime-suite mentions to the current `9`-case / `0.829x` numbers, or explicitly label the paper as intentionally using an older `11`-case snapshot and cite its provenance.

2. **[CRITICAL] The real-program validation section misreports the external-validation aggregates.** `paper.tex:141` says that on `14` unique programs, exec-time is `0.514x`; `paper.tex:515` says llvmbpf wins code size on `36/36` unique programs. The current source says `0.514x` is the geomean across `98` paired runnable instances, while the `14`-unique-program geomean is `0.484x`, and llvmbpf is smaller on `34/36` unique programs, not `36/36`: `docs/micro-bench-status.md:147`, `docs/micro-bench-status.md:164-165`, `docs/micro-bench-status.md:202`, `docs/micro-bench-status.md:497`; `docs/paper-comparison.md:430`, `docs/paper-comparison.md:446-447`. Fix: change the prose to “`0.514x` across `98` paired runnable instances” and correct the code-size row to `34/36`. If you want the unique-program exec-time number, use `0.484x`.

3. **[MAJOR] Table 8 mixes two different corpus runs and mislabels the second row as “Katran rerun.”** The Calico row in `paper.tex:1104-1106` comes from the new targeted production pass (`docs/tmp/new-corpus-v5-results.md:14-43`), but the `15/15` row in `paper.tex:1107-1109` comes from the older broader rerun (`docs/tmp/production-corpus-v5-rerun-results.md:14-31`; also summarized in `docs/kernel-jit-optimization-plan.md:726`). I could not find a Katran-only `15/15` summary in the cited artifact; the broader rerun includes many non-Katran programs with `Applied=yes` in `docs/tmp/production-corpus-v5-rerun-results.md:98-120`. As written, the table looks apples-to-apples, but it is actually mixing different corpora and scopes. Fix: either split this into two tables/subsections with explicit dataset labels and timestamps, or normalize onto a single production dataset.

4. **[MAJOR] The production-corpus subsection reports success counts and a cherry-picked example, but omits the top-line outcome that reviewers will ask for.** The new targeted production pass already has a summary: `60` measured runtime pairs, `45` programs with applied rewrites, speedup geomean `1.024x`, median `1.031x`, min `0.489x`, max `1.842x`, code-size ratio geomean `1.001x` (`docs/tmp/new-corpus-v5-results.md:27-39`). The paper instead gives only `45/45` success plus one favorable example (`calico_tc_maglev`) in `paper.tex:1104-1106`. That is incomplete enough to look selective, especially because the same source shows real regressions as well as wins. Fix: add the aggregate production numbers, and separate “overall measured set” from “programs with applied rewrites” so readers can see both coverage and effect size.

5. **[MAJOR] The paper’s headline site-count claims are discovery results, but the limitations section does not adequately disclose that the largest site reservoir is still unvalidated.** The abstract and evaluation repeatedly emphasize `514` objects / `21` new production objects / `2,146` directive sites (`paper.tex:79-80`, `paper.tex:1058-1060`). But `docs/tmp/corpus-expansion-round2.md:42-48` shows that `loxilb` alone contributes `1,827/2,146` sites, and `docs/tmp/new-corpus-v5-results.md:57-59` shows that all three `loxilb` objects had `0` compile successes in the targeted pass. The current limitations section mentions Cilium load failures (`paper.tex:1174-1177`) but not the fact that the densest newly discovered site pool is still unmeasured. Fix: explicitly state that the `2,146-site` number is an inventory/discovery result dominated by unvalidated `loxilb`, and separate “site census” from “successful recompile/runtime evaluation.”

6. **[MODERATE] The statistical story is mostly correct, but the presentation does not fully match the stated methodology.** Section 3 says significance uses paired Wilcoxon with BH correction (`paper.tex:315-317`), and the rigorous source indeed reports BH-adjusted `q` values (`docs/tmp/rigorous-benchmark-results.md:38`, `docs/tmp/rigorous-benchmark-results.md:54`, `docs/tmp/rigorous-benchmark-results.md:70`, `docs/tmp/rigorous-benchmark-results.md:86`, `docs/tmp/rigorous-benchmark-results.md:102`). The evaluation tables then report only raw `p` values in `paper.tex:980-984`. The significance directions remain correct, so this is not a correctness bug, but reviewers will reasonably ask whether the claims are corrected or not. Fix: either report `q` alongside `p`, or say explicitly that all significance statements in the table remain significant after BH correction.

7. **[MODERATE] Table 1 still contains under-explained or weakly supported rows.** The `50`-benchmark “full rerun” row in `paper.tex:345` and the runtime row in `paper.tex:346` have no CIs or significance counts, and Appendix A documents only the `31 authoritative` benchmarks (`paper.tex:1355`). This makes the `50`-benchmark row look like an orphaned intermediate result. Fix: either document the additional `19` pure-JIT cases and the runtime-suite provenance, or remove these secondary rows and keep the paper centered on the authoritative `31`-benchmark characterization plus the rigorous re-JIT evaluation.

8. **[MODERATE] Some claims need tighter caveats to avoid overclaiming the scope of the evidence.** The root-cause decomposition is based on `22` benchmarks with JIT dumps (`paper.tex:404`), but the abstract/introduction/conclusion summarize the `89%` figure as if it applied to the full suite (`paper.tex:63`, `paper.tex:155`, `paper.tex:1319-1321`). Similarly, the real-program exec-time evidence is still a lightweight packet-path validation, not a workload-level claim (`docs/micro-bench-status.md:168`, `docs/micro-bench-status.md:202`). Fix: add the `22`-benchmark caveat where the `89%` figure first appears, and soften “confirms the trends” in the abstract to something closer to “partially corroborates the direction on runnable real-program instances.”

9. **[MINOR] There are still presentation-quality issues that would distract reviewers.** `paper.tex:436-443` still contains a literal TODO/placeholder figure. `paper.tex:1271` cites `\cite{k2}` as “the llvmbpf userspace runtime itself,” but `references.bib:1-10` shows `k2` is the K2 superoptimization paper, not an llvmbpf reference. The implementation breakdown also looks numerically sloppy: `paper.tex:793-803` includes a userspace row but ends with “Total kernel ~1800,” which does not reconcile cleanly with the listed components. Fix: replace the placeholder figure, correct the llvmbpf citation, and make the implementation table arithmetic and labels precise.

## What A Top-Venue Reviewer Will Challenge

- The single strongest challenge will be empirical trust: if the runtime-suite and external-validation numbers are still stale or mislabeled, the reviewer will stop trusting the rest of the paper.
- The second challenge will be external validity: the `2,146-site` headline is compelling, but most of those new sites come from `loxilb`, which is not yet part of the successful recompile/runtime story.
- The third challenge will be production relevance: the paper needs to report the aggregate effect of the targeted production pass, not just success counts and one good example.
- The fourth challenge will be generality: the paper still has only one x86-64 microarchitecture, no end-to-end deployment result, and WIDE_MEM still lacks a positive significant latency result.

## Most Important Fixes Before Submission

1. Update every runtime-suite number to the current authoritative `9`-case / `0.829x` result, or explicitly defend using an older snapshot.
2. Correct the real-program validation prose and table to `34/36` code-size wins, `0.514x` on `98` paired runnable instances, and `0.484x` on `14` unique runnable programs.
3. Rewrite the production-corpus subsection so it does not mix incompatible datasets, and report the aggregate targeted-pass result (`60` pairs, `1.024x` geomean, `1.001x` code-size ratio) alongside the representative examples.
4. Add a limitation stating that the `2,146-site` inventory is dominated by `loxilb`, which is still not in the successful runtime-validation path.
5. Replace the placeholder figure and clean up citation/provenance issues before any external circulation.

## Writing Quality

The prose is generally clear, the story arc is strong, and the current limitations section is much better than a typical systems draft. The remaining writing problems are mostly precision problems: a few labels are misleading, a few summaries compress distinct datasets too aggressively, and a couple of presentation details still make the paper feel unfinished.
