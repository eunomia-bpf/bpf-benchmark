## Kernel Changes

File: `vendor/linux-framework/arch/arm64/net/bpf_jit_comp.c`

### 1. P0: stage subprog recompile state on every recompile pass

ARM64 now matches x86’s staged recompile bookkeeping:

- `jit_recompile_bpf_func` is published for subprogs during the first recompile pass
- `jit_recompile_staged` remains gated on `!prog->is_func || extra_pass`

This fixes the original `stage_ok(..., final_pass=false)` failure mode for multi-function recompile.

### 2. P0 follow-up: use staged subprog targets during ARM64 recompile call emission

While validating on ARM64 QEMU, the new subprog testcase exposed a second ARM64-specific problem:

- first-pass staging was now present, but ARM64 `BPF_CALL` emission still fell back to the live subprog `bpf_func`
- during recompile extra-pass emission, that produced a stale / invalid subprog target

ARM64 now resolves `BPF_PSEUDO_CALL` targets from `bpf_jit_recompile_staged_func()` during recompile extra passes, instead of relying on the live `bpf_func`.

This was necessary for the new ARM64 subprog testcase to execute correctly.

### 3. P1: `BITFIELD_EXTRACT` representability guard

ARM64 `emit_canonical_bitfield_extract_arm64()` now returns `-EINVAL` when:

- `field_width == 0`
- `field_width > width - shift`

That makes non-representable `UBFX` cases fail closed and fall back to stock lowering instead of emitting `AARCH64_BREAK_FAULT`.
