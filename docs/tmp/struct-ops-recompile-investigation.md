# struct_ops Recompile Investigation

Date: 2026-03-12

Task: `#109`

## Summary

`BPF_PROG_JIT_RECOMPILE` does not reject `BPF_PROG_TYPE_STRUCT_OPS` at syscall entry. The observed scx `EINVAL` was a symptom of a deeper x86 JIT bug plus a remaining `struct_ops` design gap:

1. The immediate kernel failure was an x86 re-JIT crash in `do_jit()` while repopulating exception-table entries from a stale `aux->extable` pointer left over from the previous JIT image.
2. After fixing that crash, live attached `struct_ops` callbacks are still not safe to re-JIT, because their invocation trampolines hardcode the old `prog->bpf_func` and are not regenerated by `BPF_PROG_JIT_RECOMPILE`.

So the current tree now:

- fixes the x86 stale-extable re-JIT bug, and
- explicitly rejects attached live `struct_ops` with `-EOPNOTSUPP` plus a diagnostic log,

but it does **not** yet provide full live `struct_ops` recompile support for `scx_rusty`.

## Source Trace

### 1. No prog-type whitelist blocks struct_ops

- `syscall.c` dispatches `BPF_PROG_JIT_RECOMPILE` directly to `bpf_prog_jit_recompile()`.
- `bpf_prog_jit_recompile()` uses `bpf_prog_get()` and does not check `prog->type`.
- `jit_directives.c` has generic `-EINVAL` checks (`flags`, `!jited`, malformed/empty policy, generic re-JIT invariants), but no `STRUCT_OPS` reject path in the original code.

Relevant files:

- `vendor/linux-framework/kernel/bpf/syscall.c`
- `vendor/linux-framework/kernel/bpf/jit_directives.c`

### 2. struct_ops bodies are JITed normally, but invocation uses separate trampolines

- The `struct_ops` program body still goes through the standard BPF JIT path (`bpf_int_jit_compile()` / verifier subprog JIT).
- However, live `struct_ops` attachment builds per-member trampolines in `bpf_struct_ops_prepare_trampoline()`.
- On x86, those trampolines are generated by `arch_prepare_bpf_trampoline()` and emit a direct call to the current `p->bpf_func` when the trampoline image is built.

Relevant files:

- `vendor/linux-framework/kernel/bpf/bpf_struct_ops.c`
- `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c`

## Actual Failure Path

The real scx failure was not an intentional `return -EINVAL;` branch in `jit_directives.c`.

Reproducing `python3 e2e/run.py scx --smoke --duration 1` in a framework-kernel VM showed repeated guest oopses:

- call chain: `bpf_prog_jit_recompile()` -> `bpf_jit_recompile_prog_images()` -> `bpf_int_jit_compile()` -> `do_jit()`
- fault site: `do_jit+0x2ac9`
- resolved with `scripts/faddr2line` to:
  - `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c:4731`
  - `ex->data = EX_TYPE_BPF;`

The bug is in the extable population block:

- during the first re-JIT pass, `image == NULL` and `rw_image == NULL`
- but `bpf_prog->aux->extable` still points to the **old** RO JIT image from the previous compile
- `do_jit()` sees non-NULL `aux->extable`, computes `ex` from that stale pointer, and then writes through it
- the write faults against the old RO image

This explains the repeated `scx` failures: the loaded callbacks carry exception-table state, so re-JIT hit the stale-extable bug before userspace received a meaningful per-program log.

## Fix Applied

### 1. x86 stale-extable fix

File:

- `vendor/linux-framework/arch/x86/net/bpf_jit_comp.c`

Change:

- only populate extable entries when `aux->extable`, `image`, and `rw_image` are all valid
- this keeps first-pass re-JIT from writing into the previous RO image

### 2. Explicit attached-struct_ops rejection

File:

- `vendor/linux-framework/kernel/bpf/jit_directives.c`

Change:

- if `prog->type == BPF_PROG_TYPE_STRUCT_OPS` and the program is already associated with a live struct_ops map, reject recompile with:
  - `-EOPNOTSUPP`
  - log: `attached struct_ops programs are not supported: associated trampoline must be regenerated`

This prevents a misleading â€śsuccessfulâ€ť body re-JIT that would still leave the live scheduler trampoline calling the stale image.

## Why Full struct_ops Support Still Needs More Work

Even after the extable crash fix, live `struct_ops` is not fully supported because:

- the current recompile path updates `prog->bpf_func`
- but the active per-member `struct_ops` trampoline still calls the old address baked into its generated image

Full support would require kernel work roughly along these lines:

1. identify the associated live struct_ops map/member(s) for the target program
2. regenerate the corresponding trampoline image(s) after re-JIT
3. atomically swap the callable pointer in the live kernel struct
4. reprotect/free old trampoline images safely, likely with RCU-style lifetime handling

That is a real kernel feature addition, not a one-line whitelist fix.

## VM Verification

### Before the fix

- `scx_rusty` smoke reproduced repeated guest oopses in `do_jit()`
- top frame resolved to `arch/x86/net/bpf_jit_comp.c:4731`

### After the fix

- `scx_rusty` smoke no longer oopses the guest
- the live recompile result is now explicit:

```text
BPF_PROG_JIT_RECOMPILE: Operation not supported
attached struct_ops programs are not supported: associated trampoline must be regenerated
```

- ordinary non-`struct_ops` recompile still works:
  - `micro/build/runner/micro_exec run-kernel --program micro/programs/cmov_select.bpf.o --repeat 1 --recompile-v5 --recompile-cmov --compile-only --io-mode staged --input-size 4104`
  - result: `"recompile":{"applied":true,...}`

## Bottom Line

The original scx `EINVAL` was not caused by a simple `prog->type` whitelist omission.

The concrete kernel bug was stale extable reuse in x86 re-JIT. That is fixed in the current tree.

But live attached `struct_ops` still need trampoline regeneration, so the correct current behavior is an explicit `-EOPNOTSUPP`, not pretending that `BPF_PROG_JIT_RECOMPILE` can already rewire a running `sched_ext` scheduler.
